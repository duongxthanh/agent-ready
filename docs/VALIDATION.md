# Validation — does the score reflect reality?

`agent-ready` computes its verdict **deterministically**: the core measures what a
no-JavaScript crawler actually receives, and the model never guesses the score. Two
questions matter for trust:

1. **Is the verdict reproducible?** Anyone should get the same number on the same input.
2. **Is the verdict true?** Does it agree with an independent reference scanner on the
   signals they both check?

This doc answers both — and everything here runs against the committed
[`demo/`](../demo/) fixture, so you can reproduce it with no network and no real site.

## 1. The fix-and-prove loop is reproducible

The [`demo/`](../demo/) folder holds two snapshots of the same fictional business: a
client-rendered SPA shell (`before/`) and the same site after applying agent-ready's
fixes (`after/`).

```bash
npm install && npm run build
node dist/cli.js ./demo/before    # before the fix
node dist/cli.js ./demo/after     # after the fix
```

| | Before (`demo/before`) | After (`demo/after`) |
|---|---|---|
| **Score** | **33 / F** | **97 / A** |
| Access | 15/15 | 15/15 |
| Permission | 7/15 | 15/15 |
| Readability | **0/35** | **35/35** |
| Understandability | 11/25 | 25/25 |
| Discoverability | 0/10 | 7/10 |

The headline is **Readability 0/35** in `before/`: the page is an empty `<div id="root">`
shell with zero words of visible text in the raw HTML. That is exactly what GPTBot /
ClaudeBot / PerplexityBot see, because they do not run JavaScript — the whole reason the
tool exists.

`after/` reaches 97, not 100, **on purpose**: the two remaining failures —
`Link` headers (RFC 8288) and markdown content negotiation — are server-level and cannot
be set by static files in a folder. The rubric correctly keeps flagging them rather than
rounding up. That honest ceiling is the point: the score isn't gamed.

Every artifact in `after/` was generated by agent-ready's own deterministic generators
(`generateRobotsTxt`, `generateSitemap`, `generateLlmsTxt`, `generateJsonLd`,
`generateMetaTags`) plus the pre-render fix from the React playbook — none of it was
hand-faked. See [`demo/README.md`](../demo/README.md).

## 2. The verdict agrees with an independent reference scanner

To check the core reports the **true state** of a site (not just an internally
consistent one), its output was cross-checked against an independent reference —
Cloudflare's [isitagentready.com](https://isitagentready.com) (`POST /api/scan`).

The two tools have **different scopes**, so only the overlap is comparable:

- **agent-ready** measures *crawler readability / AEO* — can a no-JS AI crawler read your
  content, and are the basic discovery files present.
- **isitagentready** measures the *agentic-web protocol surface* — MCP/A2A server cards,
  OAuth discovery, DNS-AID, WebMCP, x402/commerce.

They overlap on `robots.txt`, `sitemap.xml`, and homepage reachability. On those they
**agree**, including the trap below.

### The SPA-fallback false positive (the case that matters)

SPA hosts (Vite dev, Netlify, Vercel, CRA) return `index.html` with **HTTP 200,
`text/html`** for *unknown* paths. A naïve "is the file there?" check reads that as
"robots.txt present" — a false positive. agent-ready guards against it with
[`looksLikeHtmlDoc`](../src/fetch.ts): a real `robots.txt` / `sitemap.xml` / `llms.txt` is
never an HTML document, so an HTML body means the resource is absent.

When a SPA-fallback build is scanned through a tunnel, both tools reject the HTML shell as
a stand-in for `robots.txt`/`sitemap.xml` — isitagentready reports *"404, soft-404, or
HTML response"*, agent-ready reports *not found*. **They agree.** Regression test:
[`tests/fetch.test.ts`](../tests/fetch.test.ts) — *"treats SPA-fallback HTML for
robots/sitemap/llms as absent."*

### User-agent is what counts

One instructive wrinkle when cross-checking over a tunnel: a free tunnel injects a
browser-warning interstitial for non-bot user-agents. Feeding agent-ready a browser UA
(`--ua "Mozilla/5.0"`) made it receive that interstitial and report robots/sitemap fail —
**matching isitagentready's browser-UA result exactly**. The two scanners agree when given
identical bytes; apparent divergence was the tunnel serving different content per UA. The
lesson is baked into the tool: *what the AI crawler's UA receives is what gets scored.*

---

*Method: `agent-ready <url|folder> [--ua]` for agent-ready;
`curl -X POST https://isitagentready.com/api/scan -d '{"url":"<url>"}'` for the reference.
The `demo/` runs above are fully offline and reproducible; the cross-scanner agreement was
observed against live SPA hosts.*
